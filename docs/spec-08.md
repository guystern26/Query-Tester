# Spec 08 — Payload Alignment

## Unified Data Shape

State shape = save file = API payload. The `TestDefinition` interface is the single source of truth across all layers.

---

## Save File Format (Version 2)

```json
{
  "version": 2,
  "savedAt": "2025-01-15T10:30:00Z",
  "activeTestId": "uuid",
  "testDefinition": [TestDefinition],
  "payload": [],
  "testResults": TestResponse | null
}
```

- `version` discriminates file format. Only version 2 is currently supported.
- `testDefinition` is an array of TestDefinition objects (typically one).
- `payload` is reserved (empty array).
- `testResults` is optional — included when saving after a run.

---

## Import / Export (fileSlice)

- **Export:** `saveToFile()` serializes current store state to the save file format.
- **Import:** `loadFromFile()` validates version field, extracts the test definition, and loads it into the builder via store actions.
- File extension: `.json`.

---

## Bug Report Attachment

Same format as the save file. Generated by BugReportButton, includes current test state and results if available. Importable via the Import button in TopBar.

---

## Alert Email Attachment

Same save file format attached to failure notification emails. Includes `testResults` from the failed run. Recipients can import directly into the builder to reproduce the failure.

---

## Case Convention Boundary

| Layer | Convention | Example |
|-------|-----------|---------|
| Frontend (React/Zustand) | camelCase | `scenarioName`, `fieldValues` |
| API layer (`src/api/`) | Translation boundary | Converts at the edge |
| KVStore | camelCase | Frontend writes directly via REST |
| Backend internal | snake_case | `scenario_name`, `field_values` |

**Translation rules:**
- camelCase to snake_case: `payload_parser.py` converts incoming request payloads for internal backend processing.
- snake_case to camelCase: `response_builder.py` converts outgoing responses for the frontend.
- The `api/` directory is the ONLY place frontend-side translation occurs.
- `configApiMappers.ts` handles config-specific mapping between frontend camelCase and backend snake_case.

---

## KVStore Specifics

KVStore collections store data in camelCase because the frontend writes to them directly via Splunk REST endpoints. The backend reads KVStore records in camelCase and converts internally as needed.

Boolean values in KVStore are stored as strings (`"1"` / `"0"`). JavaScript `"0"` is truthy, so normalization is required:
- Backend: `_normalize_bools()` in handlers converts on GET/PUT responses.
- Frontend: `normalizeScheduledTest()` in `scheduledTestsApi.ts` as a safety net.

---

## Backend Payload Flow

```
Frontend (camelCase JSON)
  -> POST /services/splunk_query_tester/data/tester
  -> payload_parser.py (camelCase -> snake_case dataclasses)
  -> test_runner.py (processes with snake_case internally)
  -> response_builder.py (snake_case -> camelCase JSON)
  -> Frontend (camelCase response)
```

No `dataclasses.asdict()` — it produces snake_case. All backend-to-frontend serialization uses explicit `_to_dict()` methods with camelCase keys.
