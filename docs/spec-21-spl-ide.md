# spec-21 — SPL IDE (Query Development Environment)

## Overview

The SPL IDE is a second top-level view in the Query Tester Splunk app. It appears
as a native tab in Splunk's app navigation bar alongside the existing Query Tester
view. The IDE provides an interactive query development environment where users
build, analyze, and preview SPL queries.

**Key architectural decisions:**

1. **Native Splunk view** — second entry in `default.xml`, own HTML template and
   JS entry point. Navigation uses Splunk's built-in app nav bar.

2. **Reuses the Builder** — same `StartPage` component with sections hidden via a
   `mode` prop. Same store, same components, same Ace editor.

3. **Separate backend endpoint** — `POST /data/tester/ide_run`. Reuses the same
   Python execution files (query executor, SPL analyzer) but does NOT validate
   results. Returns raw results + AI analysis notes.

4. **Permissions-aware** — the IDE runs queries as the logged-in user. Users can
   only query indexes and apps their Splunk roles allow. Admin-only features
   (like Setup) remain gated.

The IDE is a **day-1 feature**.

### What Happens on Run

The IDE calls a **different endpoint**: `POST /data/tester/ide_run`.

**No validation.** The user may legitimately expect 0 results (e.g., testing a
negative filter). The backend simply executes the query and returns results.

Request:
```json
{
  "app": "search",
  "query": "index=main sourcetype=access_combined | stats count by src_ip",
  "timeRange": { "earliest": "-24h", "latest": "now" }
}
```

Response:
```json
{
  "status": "success",
  "resultCount": 42,
  "executionTimeMs": 850,
  "resultRows": [
    { "src_ip": "10.0.0.1", "count": "15" },
    { "src_ip": "10.0.0.2", "count": "27" }
  ],
  "splAnalysis": {
    "commandsUsed": ["search", "stats"],
    "unusualCommands": [],
    "unauthorizedCommands": []
  },
  "aiNotes": [
    {
      "severity": "info",
      "category": "results_insight",
      "message": "Query returned 42 results. Top contributor is src_ip 10.0.0.2 with 64% of total count.",
      "suggestion": null
    },
    {
      "severity": "warning",
      "category": "optimization",
      "message": "No time range specified — query scans all time. Consider adding earliest=-24h.",
      "suggestion": "earliest=-24h latest=now"
    }
  ],
  "warnings": [],
  "errors": []
}
```

**aiNotes** are generated by sending the query + results summary to the LLM after
execution. If no LLM is configured, `aiNotes` is empty. The `splAnalysis` and
static warnings still run without LLM.

## Permissions & Security

### User Context

The IDE runs queries using the **session user's token** — same as the Query Tester.
The user can only access indexes, apps, sourcetypes, and lookups their Splunk roles
permit. No elevation, no impersonation.

### IDE-Specific Concerns

| Concern | How It's Handled |
|---------|------------------|
| Index access | Splunk enforces via `srchIndexesAllowed` on the user's role |
| App namespace | Query runs in the selected app's context — macros, lookups, saved searches scoped accordingly |
| SPL command restrictions | Command policy (from Setup) applies — blocked commands rejected before execution |
| Save As (saved search/alert) | Uses the session user — saved searches owned by them with their permissions |
| Open in Splunk | URL opens in the user's existing Splunk session — normal auth applies |
| LLM API calls | LLM config is admin-only (Setup page). IDE users consume it but can't change it |
| Rate limiting | The `ide_run` endpoint should respect the same rate limits as the tester |

### No Admin Required

The IDE is available to any user who can access the app. The Setup page remains
admin-only. AI features work for all users if an admin has configured the LLM
endpoint.

## Backend: `POST /data/tester/ide_run`

### Handler

Wired into `query_tester.py` dispatcher:

```python
if path_info.startswith('/ide_run'):
    from ide.ide_run_handler import IdeRunHandler
    return IdeRunHandler(self.session_key).handle(method, payload)
```

### Execution Flow

```
1. Parse request (app, query, timeRange)
2. Run SPL analyzer (same spl_analyzer.py) — get warnings, commands used
3. Check command policy — reject blocked commands
4. Execute query via query_executor.py (same as tester, but no data injection)
5. Collect results (capped at 1000 rows)
6. If LLM configured: send query + results summary to LLM for aiNotes
7. Return response
```

**Reused modules:** `spl_analyzer.py`, `query_executor.py`, `command_policy`.
**NOT used:** `payload_parser.py`, `data_indexer.py`, `query_injector.py`,
`result_validator.py`, `event_generator.py`.

### AI Results Analysis (post-execution)

After getting results, if LLM is configured, the handler sends a request with:

```python
system_prompt = """You are a Splunk query results analyst. Given an SPL query,
its results summary, and optionally a previous analysis, provide brief actionable
notes. Focus on: result patterns, data quality issues, optimization suggestions,
and anything unexpected. Return ONLY a JSON array of objects with: severity
(error/warning/info), category (results_insight/optimization/data_quality/
best_practice), message (string), suggestion (string or null).
No explanation. No markdown. JSON array only."""

user_message = f"""Query: {spl}
App: {app}
Result count: {result_count}
Sample rows (first 10): {json.dumps(sample_rows)}
{f"Previous analysis: {json.dumps(prior_analysis)}" if prior_analysis else ""}"""
```

The `prior_analysis` is the latest `analyze_spl` result if one was run before the
query execution. This lets the AI correlate its field warnings with actual results
(e.g., "field 'src_ip' was flagged as a possible typo — results confirm 0 values
for this field, consider using 'src' instead").

### Result Cap

Results are capped at 1000 rows. If the query returns more, the response includes
a warning: `"Results truncated to 1000 rows. Full results available in Splunk."`.

## Backend: `POST /data/tester/analyze_spl` (unchanged from earlier spec)

Static + LLM analysis of SPL structure. Returns notes about typos, unused fields,
optimization hints. This runs **independently of query execution** — triggered by
the frontend on SPL change (debounced).

---

## Feature: Intelligence Panel (Analysis Notes)

Sits to the right of the query editor. Contains analysis notes from the
`analyze_spl` endpoint (triggered on SPL change, debounced 800ms).

Shows notes grouped by severity. Clickable notes scroll the Ace editor to the
relevant line. "Apply" button on notes with suggestions.

**This panel does NOT show AI results notes.** Those appear in the ResultsPanel
after a run.

---

## Feature: AI Results Notes (in ResultsPanel)

After a run, the ResultsPanel shows the usual results table PLUS an "AI Analysis"
section below (or as a collapsible card above the table). This section renders
the `aiNotes` from the `ide_run` response.

Same rendering as AnalysisPanel notes: severity icons, category badges, messages.
But these are contextual to the results, not the SPL structure.

---

## Feature: AI Chat / Context Input

Two possible approaches depending on complexity:

### Option A — Simple Context Bar (recommended for v1)

Instead of a full chat, add a **context text area** above or below the Ace editor
with a label: "Describe what this query should do (helps the AI give better analysis)".

This text is sent along with the `analyze_spl` and `ide_run` requests as an
additional `userContext` field. The LLM prompts include it:

```
"The user describes their intent: {userContext}"
```

This avoids the complexity of conversation history, token management, and streaming.
The user gets better AI analysis by providing context. One input, no back-and-forth.

### Option B — Full Chat (future enhancement)

Collapsible chat panel with conversation history, as originally specified.
Uses `callLLM` from `llmApi.ts` with the current SPL injected into the system
prompt. Conversation trimmed to last 10 turns.

**Decision:** Start with Option A. If users request interactive chat, add Option B
as a future enhancement. The `userContext` field is useful either way — it feeds
into analysis quality.

## Feature: Destination Actions

### Send to Query Tester

Via `localStorage` transfer key (`qt_ide_transfer`). Builder picks it up on mount.

### Open in Splunk

Opens `{splunkWebUrl}/app/{app}/search?q={encodedSpl}` in a new tab.

### Save As

Saved search / Alert / Report via `POST /splunkd/__raw/servicesNS/{user}/{app}/saved/searches`.

The `{user}` comes from the session — saves as the logged-in user, not as admin.
This is critical for permissions — users create saved searches they own.

## State

Same Zustand store. Additional fields:

```typescript
interface IdeState {
  // Analysis (from analyze_spl)
  analysisNotes: AnalysisNote[];
  analysisLoading: boolean;

  // User context for AI
  ideUserContext: string;

  // IDE run results (from ide_run)
  ideResponse: IdeRunResponse | null;
  ideRunning: boolean;

  // Actions
  setIdeUserContext: (ctx: string) => void;
  analyzeQuery: (spl: string, app: string, userContext: string) => Promise<void>;
  runIdeQuery: (spl: string, app: string, userContext: string) => Promise<void>;
  cancelIdeRun: () => void;
}
```

### Types

```typescript
interface IdeRunResponse {
  status: 'success' | 'error';
  resultCount: number;
  executionTimeMs: number;
  resultRows: Record<string, string>[];
  splAnalysis: SplAnalysis;
  aiNotes: AnalysisNote[];
  warnings: SplWarning[];
  errors: Array<{ code: string; message: string; severity: string }>;
}

interface AnalysisNote {
  id: string;
  severity: 'error' | 'warning' | 'info';
  category: string;
  message: string;
  line: number | null;
  suggestion: string | null;
}
```

## Phased Implementation

### Phase 1 — Splunk View & Mode Toggle
Splunk config, Vite entry, mode prop, hidden sections. Run button calls
existing tester endpoint temporarily (just to verify wiring).

### Phase 2 — IDE Run Endpoint
`POST /data/tester/ide_run` backend handler. No validation, returns raw results.
Frontend `ideApi.ts` + store `runIdeQuery` action. ResultsBar wired to IDE response.

### Phase 3 — Analysis Engine
`POST /data/tester/analyze_spl` backend handler with static analyzer.
AnalysisPanel frontend component. Debounced auto-analysis.

### Phase 4 — AI Integration
LLM-enhanced analysis in `analyze_spl`. Post-execution AI notes in `ide_run`.
Context input (`ContextInput.tsx`). `AiResultsCard` in results panel.

### Phase 5 — Destination Actions
Send to Tester, Open in Splunk, Save As modals. `localStorage` transfer.

### Phase 6 — Polish
Keyboard shortcuts, Ace annotations, panel persistence, error boundaries.
